LLM

Large Language Models
ampliam o que você já sabe fazer.

Modelos que entendem e geram linguagem natural com alta precisão. Não são apenas chatbots — são motores de raciocínio que leem código, planejam tarefas, chamam ferramentas e colaboram com humanos.

Agent

Agents são LLMs que
tomam ações no mundo real.

Um agent recebe uma tarefa, planeja passos, usa ferramentas (bash, editor, browser, APIs) e executa código sob a sua direção. O desenvolvedor define o objetivo — o agent faz o trabalho pesado.

LLM → recebe prompt e gera texto.
Agent → recebe objetivo e toma ações até completar.

Context & Token

O que o agent sabe e como ele pensa.

Context · Tipo 01

Contexto Negocial

  • O quê: Intenção do produto, regras de negócio, personas, jornadas, restrições de domínio.
  • Por quê importa: Um agent sem contexto negocial vai gerar código tecnicamente correto mas errado para o problema.
  • Onde vive: PRDs, briefs de produto, histórias de usuário, docs de domínio — injete no CLAUDE.md ou no início do chat.
  • Exemplo prático: "Usuários B2B nunca veem preços sem login" — sem isso no contexto, o agent vai expor preços publicamente.
Context · Tipo 02

Contexto Arquitetural

  • O quê: Stack, padrões de código, estrutura de pastas, decisões de design, convenções do time.
  • Por quê importa: Sem isso, o agent reinventa a roda — cria um novo serviço em vez de usar o existente.
  • Onde vive: CLAUDE.md, ADRs, diagramas de arquitetura, project-context.md gerado pelo BMAD.
  • Regra de ouro: Contexto negocial + arquitetural = agent que executa com a qualidade que você exige — porque foi você quem definiu os padrões.

Modos de uso

Ask, Plan, Edit, Test — cada modo tem um custo e um propósito diferente.

Token

Os 4 modos de uso

Ask
Pergunta
Use quando quer entender algo. Não gera código — gera conhecimento. Mais barato.
Plan
Planejamento
Defina a estratégia antes de escrever código. O agent pensa antes de agir.
Edit
Execução
O agent edita arquivos, cria código, refatora. Use com intenção — aqui está o maior ganho por token.
Test
Verificação
Roda testes, lê logs, verifica. O ciclo só fecha no Test — é aqui que o trabalho vira entrega.

Em todos os modos: você decide — o agent executa.

Fluxo prático

O ciclo de trabalho com agent

Ask
Entenda
→
Plan
Defina
→
Edit
Execute
→
Test
Verifique
  • Ask antes de Plan: não planeje sem entender o problema.
  • Plan antes de Edit: estratégia antes de código — o agent executa melhor com direção clara.
  • Test fecha o ciclo: "funciona na minha máquina" não é Test.
  • Repita: o ciclo é iterativo — volte ao Ask sempre que travar.

Metodologias de trabalho — o que existia e o que veio depois.

Workflow · Não Ágil

Metodologias Não Ágeis

Waterfall
  • Sequencial e linear — uma fase por vez
  • Requisitos congelados no início
  • Funciona bem em projetos com escopo fixo (ex: obras físicas)
  • Problema: software muda — waterfall não se adapta
RUP
  • Rational Unified Process — iterativo mas pesado
  • 4 fases: Inception, Elaboration, Construction, Transition
  • Muito artefato, pouco produto
  • Problema: lento demais para o mercado atual
Workflow · Ágil

Metodologias Ágeis

Fluxo contínuo Limita WIP Visualiza o trabalho Ideal para suporte e bugs
Scrum
Sprints de 1-4 semanas Daily standups Retrospectiva Planejamento iterativo
Extreme Programming

Drive Development · Métodos

TDD e BDD na prática com agents

TDD
  • Teste primeiro, implementação depois.
  • Feedback rápido a cada ciclo do agent.
  • Evita regressão e reduz retrabalho.
BDD
  • Comportamento descrito em linguagem de negócio.
  • Cenários Given/When/Then como contrato.
  • Alinha produto, QA e desenvolvimento.

TDD garante qualidade técnica; BDD garante alinhamento com valor de negócio.

Drive Development · Métodos

DDD e SDD para escalar com consistência

DDD
  • Modelo de domínio como centro da arquitetura.
  • Linguagem ubíqua entre negócio e engenharia.
  • Bounded contexts para reduzir acoplamento.
SDD
  • Especificações guiam implementação e testes.
  • Contrato explícito antes de escrever código.
  • Ótimo para delegar tarefas a múltiplos agents.

DDD organiza o domínio; SDD transforma intenção em execução verificável.

Drive Development · Ferramentas

Speckit e Kiro no fluxo de especificação

Speckit
Spec-first Padroniza requisitos Melhora handoff para agent
Kiro
Estrutura de histórias Fluxo orientado a tarefas Rastreabilidade de decisão
  • Primeiro define o que será feito, depois como.
  • Menos ambiguidade no prompt e mais previsibilidade no resultado.

Por que XP + AI?

Extreme Programming + IA

  • Pair Programming 2.0: você faz pair com o agent. Um humano, um agent, juntos no mesmo problema.
  • TDD nativo: escreva o teste primeiro, deixe o agent implementar até passar — é exatamente o que o agent faz melhor.
  • Releases pequenos: agents entregam rápido o que você especificou com precisão. Ciclos curtos dão feedback antes de acumular débito técnico.
  • Feedback do cliente: com agent, você pode prototipar e testar com usuário reais em horas, não semanas.
  • Refactoring contínuo: agents refatoram sem hesitação — delegue isso e foque na estratégia.
