Processos · Não Ágil

Waterfall

5 Vantagens
  • Documentação completa em cada fase
  • Fácil de gerenciar — uma fase por vez
  • Datas de entrega previsíveis
  • Ideal para requisitos fixos e bem definidos
  • Simples de explicar para stakeholders
5 Desvantagens
  • Inflexível para mudanças de requisitos +XP
  • Cliente só vê o produto no final +XP
  • Erros detectados tarde custam caro +XP
  • Não se adapta à complexidade de software +XP
  • Ciclos longos sem feedback real +XP
Processos · Não Ágil

RUP — Rational Unified Process

5 Vantagens
  • Iterativo — divide o projeto em fases menores
  • Orientado a risco — aborda os maiores riscos cedo
  • Altamente documentado e estruturado
  • Baseado em casos de uso — foco no usuário
  • Flexível para adaptar a diferentes contextos
5 Desvantagens
  • Extremamente pesado e burocrático +XP
  • Requer equipes grandes e treinadas +XP
  • Alto custo de implementação +XP
  • Difícil de adaptar para times pequenos +XP
  • Produz muito artefato, pouco produto +XP
Processos · Ágil

Kanban

5 Vantagens
  • Visualização clara do fluxo de trabalho
  • Limita o WIP — reduz multitarefa, aumenta foco
  • Entrega contínua sem sprints fixas
  • Baixa cerimônia — fácil de implementar
  • Melhoria incremental visível no board
5 Desvantagens
  • Sem sprints, planejamento de longo prazo é difícil
  • Pode gerar falta de priorização clara +XP
  • Não define papéis — depende da maturidade do time +XP
  • Difícil medir velocidade e previsibilidade +XP
  • Funciona mal sem cultura de WIP limit
Processos · Ágil

Scrum

5 Vantagens
  • Sprints entregam valor em ciclos curtos
  • Papéis claros: PO, Scrum Master, Dev Team
  • Feedback frequente do cliente a cada sprint
  • Backlog público — total transparência de prioridades
  • Retrospectivas melhoram o time continuamente
5 Desvantagens
  • Cerimônias podem consumir muito tempo +XP
  • Difícil de escalar para projetos grandes
  • Requer PO disponível e comprometido +XP
  • Estimativas de sprint costumam ser imprecisas +XP
  • Mudanças no meio da sprint são resistidas +XP
Processos · Ágil · Base do CodeMaster

Extreme Programming

5 Vantagens
  • Alta qualidade via TDD e pair programming
  • Feedback contínuo — cliente no processo
  • Releases frequentes e incrementais
  • Refatoração constante — código sempre limpo
  • Integração contínua previne big bang de bugs
5 Desvantagens
  • Exige alta disciplina e comprometimento
  • Pair programming difícil de justificar ao negócio
  • Documentação formal reduzida
  • Funciona melhor em equipes pequenas (<12)
  • Requer cliente dedicado e disponível sempre

Os pilares do Extreme Programming — e o que cada um significa quando o seu par é um agent.

Módulo 01 · Valores

Os 5 Valores do XP com agents

Valor No XP clássico Com agents
Comunicação
Time alinhado, sem silos
Contexto bem dado = output correto. Lixo entra, lixo sai.
Simplicidade
Menor solução que funciona agora
Quick Spec → Quick Dev. Não construa o que não foi pedido.
Feedback
Testes, builds e retrospectivas frequentes
Teste a cada ciclo, não acumule para o final.
Coragem
Refatorar sem medo, falar verdades difíceis
Iterar rápido, descartar rascunhos, recomeçar sem drama.
Respeito
Valorizar o trabalho de todos do time
Fazer code review do que o agent gerou. Ele erra.
Módulo 01 · Comunicação

O contexto é tudo

🏢
Negocial
O que o produto precisa. O problema que estamos resolvendo. O usuário afetado.
"Usuários não conseguem fechar o modal no mobile"
🏗️
Arquitetural
Como o sistema está estruturado. Quais padrões existem. O que não pode ser quebrado.
"Modais usam o componente BaseModal em src/ui/"
🎯
Da Demanda
O que precisa ser feito agora. Escopo claro. Critério de aceite definido.
"Corrigir o evento de fechar — só no iOS 16+"

"O agent é tão bom quanto o contexto que você dá."

Módulo 01 · Simplicidade

Não over-engineer

✕ Anti-patterns
✕ Pedir features que ninguém pediu
✕ Criar abstrações para uso futuro hipotético
✕ Usar BMAD completo para um bugfix simples
✕ Refatorar código que não é do escopo
✓ Patterns corretos
✓ Quick Spec → Quick Dev para demandas pequenas
✓ BMAD completo só para features grandes
✓ Entregar o mínimo que resolve o problema agora
✓ Três linhas similares > abstração prematura

As 12 Práticas

Das 12 práticas do XP, estas são as que se tornam hábito diário quando você trabalha com agents.

Módulo 02 · Práticas

As práticas que viram hábito diário

Planning Game
Montar o contexto (ESC) antes de qualquer coisa. Pergunte antes de codar.
Small Releases
Cada story deve ser entregável individualmente. Nada de "quase pronto".
TDD
Teste primeiro, implementação depois. O teste é o gatilho do ciclo — não o fim.
Pair Programming
Agora é engenheiro + agent. Você navega, ele dirige. Você sempre revisa.
Collective Ownership
O agent lê qualquer arquivo do projeto. Você também deveria conhecer todo o código.
Integração Contínua
Build + testes passando antes de qualquer merge. Sem exceções no CI.
Módulo 02 · Pair Programming

Pair programming reimaginado

Antes
Dev A
Escreve o código
Dev B
Revisa em tempo real, aponta erros, sugere melhorias
Agora
Engenheiro
Define contexto → Valida resultado → Decide arquitetura → Faz code review
Agent
Lê o código base → Executa o plano → Gera o diff → Roda os testes

Você não escreve o código — você dirige. O agent é o co-pilot que você sempre revisa.

Módulo 04 · Quick Flow

O fluxo rápido na prática

01
Quick Spec
Descreve o problema em linguagem de negócio. O agent faz perguntas, analisa o código e gera um documento com a análise + solução proposta.
02
Validar o documento
Você lê, entende e confirma que a solução faz sentido. Esse passo é seu — não pule. É aqui que você exerce o papel de PO + TL.
03
Quick Dev
Chama o Quick Dev com o documento gerado. O agent lê o documento e executa a implementação.
04
Verificar resultado
Você testa, revisa o diff e valida o comportamento esperado. Só então é done.

Done sem teste não é done. Os 4 critérios obrigatórios para qualquer entregável.

Módulo 05 · Definition of Done

O que é "done" no XP com agents

✔
Testes passando
TDD/BDD executados, zero falhas. Testes cobrem o comportamento novo e os existentes não regridem.
✔
Code review feito
O humano revisou o diff gerado pelo agent. Entende cada linha. Não aprova o que não entende.
✔
Contexto preservado
A arquitetura não foi quebrada. Padrões existentes foram respeitados. Nenhuma regressão introduzida.
✔
PR limpo
Commit semântico, branch sem conflito, CI verde. O PR conta a história do que foi feito e por quê.
